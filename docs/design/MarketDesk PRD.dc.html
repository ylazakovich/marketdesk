1 · Product overview, goals & non-goals

MarketDesk is a SaaS control panel that lets a seller manage one product catalogue and publish, monitor and optimize listings across many marketplaces from a single place. The product must feel premium and minimal — comparable in polish to Linear, Stripe Dashboard, Vercel and GitHub — prioritizing usability and whitespace over decoration.

Supported marketplaces. The platform is built to be marketplace-agnostic. Launch integrations, in connection priority: OLX, Allegro, Vinted, Facebook Marketplace (connected at launch) and eBay, Etsy, Amazon (future-ready — surfaces present, connection deferred). MarketDesk is not any one of these marketplaces; it is a management layer above them.

Goals

  • One catalogue → many marketplaces: create a product once, publish and sync everywhere.
  • Give the seller a single source of truth for revenue, profit, listing health and marketplace status.
  • Automate repetitive selling work via the Hermes AI agent (create/optimize/relist/price), always with an auditable trail and human-in-the-loop approval where money is involved.
  • Surface problems (competitor undercuts, expired listings, sync errors) before they cost sales.

Non-goals (v1)

  • No payment processing, shipping-label purchase or order fulfilment inside MarketDesk (order events are read-only references).
  • No multi-user roles/permissions beyond a single workspace owner (team management is post-v1).
  • No native mobile app; the web app is responsive but desktop-first.
Quality bar. Rounded cards (12–20px radii), soft layered shadows, generous whitespace, a single restrained accent (indigo), and no visual clutter. Every element must earn its place. Both light and dark themes are first-class and must be fully specified for each screen.

2 · Users & personas

Primary — “The power reseller” (Kamil)

Runs a small commerce operation, 100–500 active listings across 3–4 marketplaces. Time-poor, price-sensitive, checks the app several times a day. Wants to spend minutes not hours: bulk edits, fast triage of what needs attention, and trust that the AI is working while they are away. Comfortable with dashboards; impatient with clutter and latency.

Secondary — “The scaling seller”

Growing catalogue, evaluating whether to add eBay/Etsy/Amazon. Needs marketplace comparison analytics and confidence that connecting a new channel is low-effort and reversible.

Design implications

  • Optimize for scanning: strong number hierarchy, tabular figures, status colour-coding.
  • Every destructive or money-affecting action needs confirmation; every AI action needs a visible status and an undo/approve path.
  • Keyboard affordances (global search ⌘K) and bulk operations are expected, not luxuries.

4 · Design system & global components

Built on MUI Material. Component behaviour and accessibility follow MUI defaults; the tokens below theme it. Typography is Manrope (UI + numerals, with tabular-nums on all figures) and JetBrains Mono for SKUs, keys and code. The theme must be produced with MUI createTheme and applied via ThemeProvider + CssBaseline; the two palettes below map to palette.mode light/dark.

5 · Global app shell & patterns

Sidebar (250px, fixed, full height)

Brand lockup (logo mark + “MarketDesk” + “MARKETPLACE OS” eyebrow), a “MENU” group with primary nav, Settings pinned to the bottom, then the account card. Collapses to icons-only below the medium breakpoint (see §18).

Top bar (64px, sticky, translucent + blur)

  • Left: page title + one-line subtitle, contextual to the current screen.
  • Right: global search field with ⌘K hint; theme toggle (sun/moon); notifications icon with unread dot; primary New product button.

Global search (⌘K)

Opens a command palette searching products, listings and Hermes events. Debounced 200ms; grouped results; ↑/↓ to move, Enter to open, Esc to close. Empty query shows recent items and quick actions.

Theme

Light is the default. The toggle flips the whole app instantly (no reload) and persists the choice (per §17 preferences). A “System” option follows OS preference. Theme changes must re-theme MUI components and all custom surfaces together.

Notifications

Icon opens a popover list of recent events (sales, alerts, sync errors, Hermes actions) with relative timestamps and deep links. Unread dot clears on open.

Loading & error conventions (apply everywhere). Each block loads independently with a skeleton; a failed block shows an inline retry card rather than blanking the screen. Network mutations are optimistic where safe (toggles, tag add) with rollback + error toast on failure; money-affecting mutations are pessimistic (await server confirm).

6 · Screen — Dashboard

Purpose. A daily-glance overview and triage surface: how the business is doing, what the AI did, and what needs the seller now.

Layout (top → bottom, max-width 1280px, centered)

  1. Stat row — 4 KPI cards: Revenue, Profit, Active listings, Needs attention. Each: label, semantic soft icon, big tabular value, delta chip (arrow + % vs last month) and “vs last month” caption. Delta is green for positive, red for negative; for Needs attention an increase is red.
  2. Sales overview (≈62% width) — area chart of Revenue & Profit over the last 30 days, with a legend. + Marketplace overview (≈38%) — connected marketplaces as logo chip + name + listing count + a fill bar (listings/capacity), with a “Manage” link to Marketplaces.
  3. Hermes AI activity (2/3 width) — “Live” pulse badge, latest 4 events (icon, title, severity tag, one-line detail, relative time), “View all” → Hermes. + right column: Quick actions (Add product, Sync all, Run Hermes, View analytics) and Needs attention list (dot severity, product name, reason, chevron → Listing).
  4. Recent events — chronological feed (dot, text, source chip, time): orders, Hermes actions, syncs, marketplace errors.

Data / bindings

Revenue & profit totals + deltas (period = current month vs previous); active-listings count = sum of connected-marketplace listings; attention count = products in attention + expired + competitor-alert. Series: 30 daily points each for revenue and profit.

States

  • Loading: skeletons per card/chart/list.
  • Empty (new account): KPIs show 0 / “—”; charts show an empty illustration; Hermes card invites “Connect a marketplace to start”; Needs-attention shows “All clear”.
  • Error: per-block retry card.

Interactions

  • Quick-action cards and links route to the relevant screen/wizard.
  • Attention items and marketplace rows are clickable (open Listing / Marketplaces).
  • The Live badge pulses only while a background Hermes run is active.

7 · Screen — Products

Purpose. Manage the whole catalogue: find, filter, sort, inspect, bulk-edit and create products. A table/card hybrid dense enough to scan hundreds of rows.

Toolbar

  • Status tabs (segmented): All, Active, Attention, Drafts — each with a live count chip. Selecting filters the list.
  • Search (name/SKU/tag, debounced 200ms), Filters (marketplace, price range, margin, tags, date), Sort (name, price, profit, margin, views, updated — asc/desc), and a list / card view toggle.

Row / card fields

Select checkbox · image (thumbnail; falls back to a coloured initials tile) · name + SKU (mono) · status badge · marketplace count · cost · selling price · profit + margin% · tags · last updated · row overflow menu.

Bulk actions

Selecting ≥1 row reveals a sticky accent action bar: “N selected”, then Add tag, Edit price, Export, Delete, and Clear. Header checkbox selects all / indeterminate for partial. Delete opens a destructive confirm modal; Edit price opens a bulk-price modal (set absolute, or +/‑ by amount/percent, with a live preview of resulting margins and a warning if any would go negative).

Footer

“Showing N products” + pagination (prev, numbered pages, next). Default page size 25; server-side paginate + sort + filter.

States

  • Empty catalogue: empty state with “Create your first product” CTA → wizard.
  • Empty filter result: “No products match” + “Clear filters”.
  • Loading: 8 skeleton rows. Row busy: spinner in the affected cell during inline update.

Interactions / validation

  • Clicking the product cell opens Listing Details; the checkbox and overflow menu do not navigate.
  • Sort state is reflected in the column header (arrow) and the URL query.
  • Bulk price edit: reject non-numeric / negative selling price; warn (don’t block) on sub-cost prices.

8 · Screen — Listing Details

Purpose. The single-product hub: everything about one product and its listings across marketplaces, plus AI recommendations and a full audit trail.

Layout — two columns

Left (main):

  • Image gallery — large cover + thumbnail strip; drag to reorder; first image is the cover.
  • Description — rich text with an Edit affordance; tags below.
  • Marketplace status — one row per connected marketplace: logo, name, per-marketplace price, views, and a Live/Error status badge.
  • Price history — sparkline/area of price over time with the current price called out.
  • Timeline — vertical, newest first: sales, Hermes price changes, syncs, creation. Node colour encodes source/severity.

Right (sticky sidebar):

  • Price/summary card — status badge, big selling price, cost / profit / margin line, primary “Edit listing” + “Share”.
  • Statistics — Total views, Watchers, Messages, Conversion, each with a delta.
  • Hermes recommends — actionable suggestion cards (e.g. add photos, enable auto-relist, bundle) each with Apply/Dismiss.
  • Activity log — who/what/when (human + Hermes actors with avatars/initials).

States

  • Draft: marketplace rows show “Not published”; primary CTA becomes “Publish”.
  • Attention: banner summarizing the issue (competitor undercut / expired) with a one-click resolve.
  • Loading / error: per-block skeleton / retry.

Interactions

  • Back link returns to Products preserving the prior filter/sort.
  • Applying a Hermes recommendation writes to the timeline + activity log and toasts with Undo.
  • Edit listing opens the same field set as the wizard’s Basic info + Pricing steps in a modal/drawer.

9 · Screen — Analytics

Purpose. Professional performance reporting across the business and per marketplace. Read-only, exportable.

Layout

  1. KPI row — Revenue, Profit, Total views, Avg. conversion, each with delta vs previous period.
  2. Revenue & profit — large dual-series area chart over the selected range.
  3. Three-up — Views (bar), Conversion rate (area), Revenue-by-marketplace (donut + legend with %).
  4. Best performing & Worst performing — ranked product lists (rank, name, units sold, revenue, trend%).
  5. Marketplace comparison — table: marketplace, listings, revenue, conversion, share (bar + %).

Controls

Global date-range selector (7d / 30d / 90d / YTD / custom) and a marketplace filter apply to every chart on the screen. A Export action (CSV + PNG of charts).

States

  • Insufficient data: charts show a “Not enough data yet” placeholder with the minimum period needed.
  • Loading: skeleton charts. Error: per-chart retry.

Interactions

  • Hovering a chart shows a tabular tooltip (date, series values in zł).
  • Best/worst product rows link to Listing Details.
  • Changing range/filter updates all charts together and reflects in the URL.

10 · Screen — Hermes AI Activity

Purpose. A complete, auditable stream of everything the Hermes agent does or proposes, with human-in-the-loop approval for anything sensitive.

Layout

  1. Agent banner (accent gradient) — Hermes identity, an “Active” pulse badge, a one-line status (“monitoring N listings across M marketplaces”), and a “Configure” button → Settings › Hermes AI.
  2. Stat row — Actions today, Awaiting review, Listings created, Time saved.
  3. Filter bar — segmented All / Suggestions / Alerts / Completed, plus a Filter (type, marketplace, product, date).
  4. Event list — one card per event.

Event card anatomy

Severity-tinted icon · title · type tag · timestamp (with clock icon) · detail sentence · “Product: name” · status indicator (dot + label). Action buttons depend on status: items needing action / pending review show Approve (primary) + Dismiss; completed/applied items show View.

Copy tone

Hermes is a proactive, concise, confident assistant: plain language, no jargon, always states the “why” and the projected impact (“views up 40%, 0 sales in 7 days → suggest −20 zł”). It never sounds uncertain or chatty.

Interactions & states

  • Approve executes the proposed change (e.g. apply new price/title), moves the event to Completed, writes to the product’s timeline, and toasts with Undo.
  • Dismiss marks the suggestion rejected (kept in history, filterable) and optionally teaches Hermes (“don’t suggest price drops on this product”).
  • Empty: “Hermes hasn’t done anything yet” + enable-autonomy hint. Loading: 6 skeleton cards.
  • New events can arrive live (prepend with a subtle highlight); the sidebar badge count = events awaiting action.

11 · Screen — Product Creation Wizard

Purpose. Create and publish a product in six guided steps. Centered, focused layout (max-width ≈820px) with a step indicator, a content card, and a Back / progress / Continue footer.

Step indicator

Six numbered nodes with connectors: Photos → Basic info → Pricing → Category → Marketplaces → Preview. Completed steps show a check and are clickable to jump back; the current step is accent-ringed; future steps are muted. Footer shows “Step N of 6”.

12 · Screen — Marketplace Management

Purpose. Connect, monitor, configure and disconnect marketplaces. Future-ready: unconnected channels are shown as first-class “Connect” cards.

Layout

  • Summary bar — “X of 7 connected”, total active listings, last full sync time, and a Sync all action.
  • Card grid — one card per marketplace (3-up, responsive).

Card — connected

Logo chip + name + status (Synced / Syncing… / N errors) with a status dot (pulsing while syncing) · a connection Switch · rows for Listings, Synchronization (Auto), Errors (green “No errors” or red count), Last sync · a Configure button.

Card — not connected (eBay, Etsy, Amazon)

Muted surface, off switch, a one-line invitation, and a primary Connect button (starts OAuth/credentials flow in a modal).

Configure drawer

Per-marketplace settings: sync frequency (real-time / hourly / manual), price rules (markup/rounding per channel), category mapping, default shipping profile, and a “Disconnect” (destructive, confirm) that unpublishes or leaves listings per user choice.

States / interactions

  • Toggling the switch off asks to confirm and pauses sync (listings retained).
  • Error count links to a per-marketplace error list (what failed, why, retry).
  • Sync all disables the button and shows per-card syncing until complete; a toast summarizes results.

13 · Screen — Settings

Purpose. Workspace and account configuration. Two-column: a left settings nav (sticky) + a right content panel. Eight sections.

13.1 General

Workspace name, default currency (PLN), timezone (Europe/Warsaw), language. Save / Cancel with dirty-state tracking; Save disabled until changed.

13.2 Hermes AI

Autonomy level (single-select cards): Suggest only · Balanced (auto safe tasks, ask on pricing) · Full auto. Plus toggles: Auto-create listings, Auto-adjust pricing, Auto-relist expired, Smart title & SEO. Optional guardrails: max auto price change %, min margin floor.

13.3 Notifications

A matrix: rows = events (New sale, Competitor price change, Listing needs attention, Sync errors, Weekly report); columns = Email, Push, Telegram; each cell a checkbox. Independent per channel.

13.4 Marketplace Accounts

List of linked seller accounts (logo, name, account handle, status badge, Manage). Reuses connection state from §12; Manage opens the same configure/disconnect flow.

13.5 API Keys

List of keys (name, masked key in mono, created date, Copy). Generate key creates a new key shown once in a modal (“copy now — you won’t see it again”); each key has Revoke (destructive confirm).

13.6 Appearance

Theme selector (Light / Dark / System) with visual previews; accent-colour swatches (indigo default + alternatives); density (comfortable / compact). Changes apply live and persist.

13.7 Telegram

Connection status; Bot handle, Chat ID; per-type toggles (Sale notifications, Daily summary, Hermes suggestions). Connect/disconnect flow with a test-message action.

13.8 Security

Change password; Two-factor authentication (toggle + status; authenticator setup flow); Active sessions (device, location, “Sign out all” — destructive). Each row is an independent action with its own confirm where destructive.

Settings-wide rules. Every panel tracks a dirty state; navigating away with unsaved changes prompts to save/discard. Destructive actions (revoke key, disconnect, sign out all, delete) always confirm. Toggles are optimistic with rollback on failure.

14 · Hermes AI event model

Every Hermes action is an event with a consistent shape so the activity feed, notifications and timelines can render uniformly.

Event types & defaults

Updated descriptionSuccessCompletedView

Suggested better titleInfoPending reviewApprove · Dismiss

Competitor price detectedCriticalAction neededMatch price · Ignore

Fields

  • id, type, severity (info · success · warning · critical), status (completed · applied · pending review · action needed · dismissed).
  • title, detail (with projected impact), timestamp, productId / product name (or “N products” for bulk).
  • proposedChange (typed payload: e.g. { field:'price', from, to }) so Approve can execute deterministically and Undo can revert.

Severity → colour mapping reuses the semantic tokens (§4.1): success/green, info/blue, warning/amber, critical/red — each with its weak tint behind the icon/tag.

Autonomy setting (§13.2) governs which types auto-execute vs land as “pending review”: Suggest only → all become suggestions; Balanced → safe types (create/description/title/relist) auto-run, pricing waits for approval; Full auto → all auto-run within the configured price/margin guardrails, still logged.

15 · Core user flows

15.1 Create & publish a product

New product (top bar / quick action) → Wizard steps 1–6 with per-step validation (§11) → Publish → per-marketplace publish progress → success toast → Listing Details. Failure on any channel keeps successes and offers retry per failed channel. Abandoning saves a Draft.

15.2 Connect a marketplace

Marketplaces → Connect on an unconnected card → auth modal (OAuth or credentials) → on success card flips to connected, initial sync runs (status “Syncing…”), summary bar count increments, and a “Connected” toast appears. Failure shows an inline error with retry.

15.3 Approve a Hermes suggestion

Sidebar badge / Dashboard Hermes card / Hermes screen → open event → Approve → change executes (pessimistic for pricing) → event → Completed, product timeline + activity log updated, badge decrements, toast with Undo (revert via proposedChange).

15.4 Resolve a competitor price alert

Critical event surfaces on Dashboard “Needs attention” and Hermes feed → Match price (applies suggested price to the affected marketplace, respecting the margin floor; blocks + warns if it would breach the floor) or Ignore → status resolves; the product’s attention flag clears if no other issues remain.

15.5 Bulk edit prices

Products → filter → select rows → bulk bar → Edit price → set absolute or ±amount/percent with a live margin preview → Apply → optimistic row updates, error toast + rollback for any that fail, success summary toast.

15.6 Switch theme

Top-bar toggle or Settings › Appearance → whole app re-themes instantly → choice persisted; “System” keeps following OS changes.

18 · States, edge cases & acceptance criteria

Cross-cutting states (must exist for every data surface)

  • Loading — block-level skeletons; never a blank screen or layout shift.
  • Empty — purposeful empty state with a single primary CTA.
  • Error — inline retry scoped to the failed block; global outage → full-screen retry.
  • Offline — banner; queue safe mutations, disable money-affecting ones.

Responsive behaviour

  • ≥1200px: full layout as specified (content capped ~1280px).
  • 768–1199px: sidebar collapses to icons; multi-column grids reflow to fewer columns; tables allow horizontal scroll within their card (never the page). All fixed-content grids use minmax(0,1fr) tracks so cards shrink rather than overflow.
  • <768px: sidebar becomes a drawer; cards stack single-column; top-bar search collapses to an icon.

Accessibility

  • WCAG AA contrast in both themes; visible focus rings; full keyboard operability (tabs, wizard, menus, modals with focus trap + Esc).
  • Status is never conveyed by colour alone — always paired with a label/icon.
  • Charts have accessible summaries; icons that act as buttons have labels.

Acceptance criteria (samples)

  • Switching any sidebar item updates only the content region in <100ms perceived; the active item and page title update accordingly.
  • Theme toggle re-themes MUI components and custom surfaces together with no flash of the wrong theme, and the choice survives reload.
  • Selecting products reveals the bulk bar with an accurate count; header checkbox shows indeterminate for partial selection; Clear deselects all.
  • The wizard blocks Continue until the current step validates, auto-saves a draft per step, and never loses input on Back.
  • Approving a Hermes pricing suggestion updates the listing, logs it, decrements the sidebar badge, and offers a working Undo.
  • Marketplaces shows 4 connected + 3 connectable cards; unconnected cards expose only Connect; no horizontal page scroll at any supported width.
  • All monetary values render in PLN with grouped thousands and tabular alignment.
